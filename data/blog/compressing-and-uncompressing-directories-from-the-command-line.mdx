Recently, I have found the need to compress directories for upload to Canvas, an
open-source learning management system (LMS) used for managing content. It
became a real pain to have to leave Terminal to get the compressing done, so I
decided to learn how to do it directly from the Command Line.

## Excluding directories from zip compression

The problem with some of my projects is that I have directories I do not want to
include, such as the `node_modules` directory. So this is what I did to exclude
`node_modules`:

```shell
zip -r stripe-dropdown.zip stripe-dropdown/* -x "stripe-dropdown/node_modules/*"
```

The command means that I am zipping recursively everything in the
`stripe-dropdown` directory, including subdirectories and files, but excluding
the `node_modules` directory and everything inside of it. I have to add the full
path to the `node_modules` directory in order to successfully ignore it. `-r`
stands for "recursively", and the `*` stands for "wildcard", meaning
"everything". The `-x` flag means "exclude", followed by the path to the
directory to exclude. I was in the parent directory of `stripe-dropdown` when I
executed this `command`, so the `path` to `node_modules` in `stripe-dropdown`
relative to its parent directory, `development`, was
`"stripe-dropddown/node_modules/*"`. On macOS, you have to encapsulate the path
to the directory to ignore with "quotes".

When unzipping, be sure to be in the same parent directory as the `.zip`. For
example, I moved mine onto the desktop so as not to confuse it with my original
and complete directory in `~/development`. Then I made sure that I changed into
the desktop directory, `cd ~/desktop`, and then executed the `unzip` command
followed by the name of the directory (stripe-dropdown) that had been zipped. No
need for adding the `.zip` extension. No more need to leave the Command Line and
go to my Mac's menu bar, searching for `compress stripe-dropdown` (or whatever
the name of the directory I want to compress is) under the file tab, and no more
need to leave Command Line to double click on a `.zip` to expand it into a
directory. It is a great time saver!

## Using tar to archive directories on macOS

tar is a file archiver program which combines multiple files into a single
archive file. This file is referred to as a tarball. A tar file differs somewhat
from a zip file in that .zip compresses files and folders to create space,
whereas the primary purpose of a tar file is to archive files and or directories
into one single file for easier storage and transfer from one computer to
another. I can even transfer a tar file to an iPhone using AirDrop from my
macOS. It alone does not compress by default. It can work in tandem with
compression tools such as gzip or bzip2 to achieve compression.

To archive a directory with tar, I run the following in Terminal:

```shell
# this is an actual project directory I archived with tar
tar -czf wikipedia_viewer.tgz wikipedia-viewer
```

This resulted in:

```shell
wikipedia_viewer.tgz
```

The .tgz extension means that I both archived and compressed the
wikipedia-viewer directory.

In order to unarchive/uncompress `wikipedia_viewer.tgz`, I would use

```shell
 tar -xzvf wikipedia_viewer.tgz
```

But let's see how much compression takes place when I archive a directory using
tar and compress it at the same time with gzip. I can run the command
`du -sh wikipedia_viewer.tgz` and I get back:

```shell
du -sh wikipedia_viewer.tgz
5.0M	wikipedia_viewer.tgz
```

That's 5MB for the compressed archive.

If I run the same command on the uncompressed, un-archived wikipedia-viewer
directory, I get back:

```shell
du -sh wikipedia-viewer
9.0M	wikipedia-viewer
```

This demonstrates a significant difference in the directory's compressed size
and uncompressed size.

Let's see what the difference would be if I zipped the wikipedia-viewer
directory:

```shell
du -sh wikipedia-viewer.zip
220K	wikipedia-viewer.zip
```

The difference is even more significant here.

## Archiving a directory with tar without compression

If, on the other hand, I want to archive a directory using tar without
compression, I would use a slightly different command:

```shell
tar cf wikipedia-viewer.tar wikipedia-viewer
```

Then, if I wanted to see the size of this tar file, I could run:

```shell
du -sh wikipedia-viewer.tar
```

Which would return:

```shell
du -sh wikipedia-viewer.tar
8.2M	wikipedia-viewer.tar
```

There's a small difference, but it's not from compression — tar without -z
doesn't compress anything. That gap is likely just overhead from how the
directory's contents are stored on disk versus bundled into one file.

## Zip vs tar and gzip

If you need something to be compressed as much as possible because you are
sending it as an email attachment, and you have a relatively small file-size
limit, then a zip file is the way to go. If, however, you are able to transfer
your archive file via AirDrop, for instance, then tar + gzip are the way to go.
Technically there is no file limit when transferring a file over AirDrop, but
large folders and certain file types could cause problems. Another reason to
choose tar + gzip over zip is because tar + gzip preserve Unix file permissions,
making them more attractive to users in Unix-like environments. Zip, on the
other hand, does not preserve file permissions by default. They have to be
specially configured. If they are not,
[file permissions are often reset to a default: 644 for files and 755 for directories](https://linuxvox.com/blog/maintain-file-and-folder-permissions-inside-archives/#:~:text=Standard%20ZIP%20tools,their%20original%20settings.).
This does not bode well for software distribution, where retaining file
permissions can be crucial.

## Conclusion

Will it be zip or tar and gzip? It depends on what you are trying to achieve. If
extreme compression is all that is important, then zip is the way to go. But if
preserving metadata such as file permissions is the end goal, then tar and gzip
should be your top choice!
