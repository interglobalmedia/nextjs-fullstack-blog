## The importance of npm audit regarding outdated JavaScript tooling, vulnerabilities and security

`npm audit` is a security tool which was introduced in `npm 6` and is integrated
into npm 6 and later.

The command line command `npm audit` first appeared in our terminal windows with
the release of `npm 6` in April 2018. When I ran `npm audit` in `npm 6`, it
scanned my Node.js project dependencies for known vulnerabilities and, if
present, identified them. This way it pointed out potential security risks in
the project.

I have written about this topic before, but feel the need to re-address it.

[Node Security/npm audit](https://docs.npmjs.com/auditing-package-dependencies-for-security-vulnerabilities)
appeared for the first time in my Terminal window in April of last year (2018).
I had read that it was a new feature in npm created by Node Security[^1], and
was excited about the prospect of knowing more about what went on under the hood
with my npm packages and how that boded for "web/software development" and
"individual machine security". Previously, many of us were under the false
impression that everything in our code and workflows using third party plugins
and packages passed muster. But the minute `npm audit` came along and we
downloaded npm packages and added libraries and frameworks using those packages,
if there were "known" vulnerabilities, Node Security (now part of npmjs) would
print out something like the following:

The first time I encountered `npm audit`, I was in a full stack JS class, and we
were told to clone a repository which had been created TWO YEARS before. This
happened right after npm6 had been released. We all did so, and I received a
message in Terminal from Node Security that there were approximately 1,000
vulnerabilities in the project. I mentioned it to the teacher, but he looked at
me as if I had three heads. That was a little over a year ago. April 2018, to be
exact.

You know what? It seems that there are still people in "teaching" positions who
don't bother to check their repositories, even though, most probably, GitHub is
sending them security alerts all the time. I know I do.

This post is for those who might be new to the JavaScript Developer ecosystem,
and might see similar warnings when cloning or downloading courses they have
paid for and from which they want to learn the best and latest practices.
Apparently, many teachers don't bother to share this kind of information because
it just is "not relevant" to their subject matter. You'd better bet your bottom
dollar it is. It is relevant to everything JavaScript, and everything all other
programming languages too.

If, when you clone or download something from a GitHub repository, for example,
and then run `npm i` (assuming there is a `package.json` present), followed by a
message stating that vulnerabilities have been detected in your package(s),
follow the instructions on running `npm audit`.

Oftentimes, there is already a remediation/patch/package update that removes the
vulnerabilities. Follow the instructions you receive in Terminal. Sometimes
there is no remediation yet, or it might be difficult to do manually. That's
when you should contact the creator/maintainer of the repository or package.

Sometimes things take a little while. After all, these packages are open source.
Sometimes, however, it is not a matter of fixing the vulnerabilities in a
current version of a package by simply updating to the latest version, but
updating one that results in breaking changes. That's when things get a bit
ugly.

Sometimes developers are too lazy (or don't have the time) to do a complete
overhaul of their workflows associated with those breaking changes, and this
results in vulnerabilities in the current package version(s) that won't go away.
This is especially NOT COOL when it happens within course code that you have
paid good money for. If the creator of the course has not updated the
code/curriculum in more than 6 months or even 3 months (things these days are
changing daily), think twice before signing up for it. Do a bit of detective
work regarding the `dependencies` and `devDependencies` in the `package.json`
before making your final decision. Sometimes, of course, we don't have access to
this information until we have actually paid for it. I don't have an answer for
that.

Be sure to do the same in your own open source repositories. It is hard to keep
up with GitHub security alerts, but just do your best to do it when you can. The
longer you wait, the more complicated the remediation.

## Node security after the Node Security Platform shutdown

Post Node Security, Node.js integrated security features directly into npm,
which includes `npm audit` and vulnerability scanning. This was meant to
strengthen the security of JavaScript applications and provide developers with
essential tools to effectively manage vulnerabilities.

Every npm package install request (`npm i`) is followed by an automatic review
against the GitHub advisory database[^2]. I also include linter tools such as
ESLint in my development workflow to identify problems in my code including
potential security vulnerabilities.

### Best practices for bolstering security with npm post Node Security

1. We should conduct regular code reviews. Conducting regular code reviews can
   help to catch code vulnerabilities and improve code quality.[^3]
1. We should update our npm packages often to make sure we have the latest
   security fixes. I am constantly checking for updates in my active
   repositories.
1. We should only include well-maintained packages that are also widely used.
   This helps minimize the risk of vulnerabilities. If I see that a package is
   not continually maintained, I tend to shy away from it. A while back I was
   looking into packages that implemented static tweets, and I came across the
   `react-tweet` npm package maintained by Vercel. It didn't look like it was
   frequently maintained, so I passed on it.

## Conclusion

In this post, I discuss the importance of `npm audit` in relation to outdated
JavaScript tooling used in development workflows and how it helps mitigate the
risk of using or committing vulnerable code. I also discuss how important it is
to learn about security tools early on in software development education. We
should adopt good habits and best practices from the start. In addition, the
integration of the Node Security Platform into the npm ecosystem resulted in a
stronger and more secure npm framework. Utilizing its built-in tools has allowed
developers like myself to proactively manage vulnerabilities in our code thereby
ensuring coding best practices.
